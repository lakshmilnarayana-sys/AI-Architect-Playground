# StreamFlix Platform Phase 4 — Backstage Software Catalog (Design)

**Date:** 2026-06-25
**Status:** Approved (design) — proceeding to plan
**Builds on:** Phases 1–3 (cluster + 35 services + observability + alerting + live incident loop). Single-source-of-truth principle continues: the catalog is generated from `graph/`.

## 1. Goal

Stand up a real **Backstage** software catalog on the kind cluster, with the StreamFlix
software graph (services, teams, people, dependencies, ownership) generated from the same
`graph/` CSVs that drive everything else, and each service linked to its Prometheus/Grafana
signals and runbooks. "The catalog IS the graph."

## 2. Existing assets mapped to Backstage entities

| Graph | Count | Backstage entity |
|---|---|---|
| `service:*` nodes | 35 | `Component` (type `service`) |
| `team:*` nodes | 13 | `Group` (type `team`) |
| `person:*` nodes | 12 | `User` |
| `DEPENDS_ON` edges | 70 | Component `spec.dependsOn` |
| `OWNS_SERVICE` edges | 10 | Component `spec.owner` (→ Group) |
| `MEMBER_OF` edges | 12 | User `spec.memberOf` (→ Group) |
| `platform/runbooks/*.md` | 8 | per-service runbook annotation/links |

A single `System` (`streamflix`) groups all components.

## 3. Decisions (confirmed; defaults chosen autonomously)

| Decision | Choice | Why |
|---|---|---|
| Run real Backstage | Yes, in-cluster on kind | "real on my kubernetes" |
| Build strategy | **Entirely inside a `node:20` multi-stage Docker image** (scaffold via `npx @backstage/create-app` at build time, inject our app-config + catalog loader, build backend) | Host has Node 25 (unsupported by Backstage) + no yarn; in-container build is reproducible and host-independent. Nothing huge committed (only Dockerfile + config + generator + generated catalog). |
| Database | **SQLite (better-sqlite3)** | Single pod, no separate Postgres; fine for a demo. |
| Auth | **Guest access** | Local demo; no SSO. |
| Catalog source | Generated entity YAML mounted via ConfigMap; Backstage `catalog.locations` type `file` | Single-source-of-truth from the graph, like the Phase-1 manifest generator. |
| TechDocs | **Runbook + dashboard LINKS via annotations** (not a full mkdocs TechDocs pipeline) | YAGNI for a demo; links are real and useful. |

## 4. Architecture

```
graph/*.csv ──generate_catalog.py──▶ catalog/ (System + 13 Groups + 12 Users + 35 Components)
   └─ mounted as ConfigMap ─▶ Backstage (in-cluster, SQLite, guest auth)
        catalog.locations: file:/catalog/*.yaml ─▶ catalog API /api/catalog/entities
```

## 5. Components

### 5.1 Catalog generator — `platform/backstage/generate_catalog.py`

Reads `graph/nodes.csv` + `graph/edges.csv` and emits Backstage entity YAML into
`platform/backstage/catalog/`:
- `system.yaml` — `System: streamflix`.
- `groups.yaml` — one `Group` per `team:*` (displayName from node name; `spec.type: team`,
  `spec.children: []`). 
- `users.yaml` — one `User` per `person:*` (`spec.memberOf` from `MEMBER_OF` edges; profile
  displayName + the role from the node description).
- `components.yaml` — one `Component` per `service:*`:
  - `metadata.name` = the k8s service name (`<short>-service`), matching everything else.
  - `spec.type: service`, `spec.lifecycle: production`, `spec.system: streamflix`.
  - `spec.owner` = the owning Group from `OWNS_SERVICE`; **default `platform-engineering`**
    for the 25 services with no ownership edge (Backstage requires an owner).
  - `spec.dependsOn` = `component:<dep-service>` for each `DEPENDS_ON` edge.
  - `metadata.annotations`: `prometheus.io/service` (the metric `service` label),
    `streamflix.io/grafana` (Grafana explore link), `streamflix.io/runbook` (the matching
    `platform/runbooks/*.md` when the service has a known failure mode), and
    `backstage.io/managed-by-location`.
  - `metadata.tags`: `customer-facing`/`internal` tier.

CLI: `python generate_catalog.py --out platform/backstage/catalog`. All entity names are
Backstage-valid (lowercase, dashes). Output is git-committed (it's the catalog snapshot)
**and** regenerated by `make backstage`.

### 5.2 Backstage app — `platform/backstage/`

- `Dockerfile` (multi-stage, `node:20-bookworm`): stage 1 scaffolds a Backstage app via a
  pinned `npx @backstage/create-app@<ver>` (non-interactive), overlays our `app-config.yaml`
  (SQLite, guest auth, `catalog.locations` → `/catalog`, allowed kinds), and builds the
  backend (`yarn install` + `yarn build:backend`); stage 2 runs the built backend on `:7007`
  serving the frontend bundle. Image `localhost:5001/streamflix-backstage:dev`.
- `app-config.yaml`: `backend.database.client: better-sqlite3` (file on emptyDir);
  `auth.providers.guest`; `catalog.locations` reading `/catalog/*.yaml` (type `file`);
  `catalog.rules` allowing `Component, System, Group, User, Domain, Resource, Location`;
  `app.baseUrl`/`backend.baseUrl` for port-forward.

### 5.3 Deploy — `platform/backstage/k8s.yaml`

Deployment + Service in a new `backstage` namespace (`backstage:7007`), with the generated
catalog mounted from a `backstage-catalog` ConfigMap at `/catalog`. Resource requests sized
for a Node backend (request 256Mi/250m, limit 1Gi/1).

### 5.4 Makefile / UX

- `make backstage` — generate catalog → create/update the `backstage-catalog` ConfigMap →
  build + `kind load` the image → apply manifests → rollout.
- `make backstage-up` — port-forward `backstage:7007` and print the URL.
- `make backstage-verify` — show the pod + the `curl /api/catalog/entities?filter=kind=component`
  count.

## 6. Acceptance criteria (the "real catalog" test)

1. `python generate_catalog.py` produces a System, 13 Groups, 12 Users, 35 Components; the
   catalog passes a structural validation (every Component has an owner + valid name; every
   `dependsOn` target exists; every User `memberOf` references a real Group).
2. Backstage pod Running in `backstage` ns.
3. `GET /api/catalog/entities?filter=kind=component` returns 35 components; a spot-check
   (`billing-service`) shows `spec.owner = group:default/billing-platform`,
   `spec.dependsOn` includes its real downstreams, and the prometheus/runbook annotations.
4. `GET /api/catalog/entities?filter=kind=group` returns 13; `kind=user` returns 12.
5. The Backstage UI (port-forwarded) renders the catalog and a component's dependency graph.

## 7. Non-goals (Phase 4)

No Postgres (SQLite), no SSO (guest), no full TechDocs mkdocs pipeline (runbook links only),
no Kubernetes plugin / live-pod view inside Backstage (the cluster is already observable via
Grafana), no scaffolder templates. The agent/incident loop is unchanged.

## 8. Risks & mitigations

- **Backstage in-container build is heavy/fragile** (`create-app` + `yarn install` pull
  ~1.5GB; build is minutes). Mitigation: pin the create-app version; build on `node:20`
  (not host Node 25); generous timeouts. **Honest fallback:** if the full Backstage build is
  infeasible in this environment, deploy a minimal catalog-API server that serves the SAME
  generated Backstage-format entities at `/api/catalog/entities` so the catalog is still real
  and queryable — documented, not hidden (mirrors the Phase-1 `node_pressure` best-effort and
  the OTel-trace honesty flags).
- **Backstage requires an owner per Component** → unowned services default to
  `group:platform-engineering`; documented.
- **Entity name validity** → generator lowercases/validates names; a test asserts every
  `dependsOn` target resolves and every Component name is Backstage-valid.
- **Catalog/graph drift** → generator is the only writer of catalog YAML; `make backstage`
  always regenerates.
