# eBPF Profiling Setup

PerfAgent supports language-independent profiling with Linux `perf`/eBPF-style tools. This path does not require Go pprof endpoints, JVM agents, Python hooks, Node flags, or code instrumentation.

## What PerfAgent Captures

When `perf` is available, `profile run --mode ebpf` captures and post-processes:

- `captured/perf.data`: raw Linux perf recording
- `captured/perf.script`: textual stack output from `perf script`
- `captured/perf.folded`: folded stacks for flamegraph tooling
- `captured/perf-flamegraph.svg`: deterministic SVG flamegraph generated by PerfAgent
- `logs/profile-*.stdout.log` and `logs/profile-*.stderr.log`: profiler command logs
- `profile-result.json`: command status, warnings, generated artifacts, and parsed top functions

The interactive HTML report includes a Flamegraph Viewer section for SVG flamegraph artifacts, with a function-name search over parsed top functions when summaries are available.

PerfAgent also plans optional eBPF profiler commands for `bpftrace`, Pyroscope eBPF, and Parca Agent when those binaries are present.

## Linux Host Requirements

The target host needs:

- Linux kernel support for `perf_event_open` and user stack capture
- `perf` installed
- debug/tracing filesystems mounted at `/sys/kernel/debug` and `/sys/kernel/tracing`
- permissions to inspect the target process
- frame pointers or usable unwind information for high-quality stack names

Useful host checks:

```bash
perf --version
cat /proc/sys/kernel/perf_event_paranoid
mount | grep -E 'debugfs|tracefs'
```

Common lab settings:

```bash
sudo sysctl kernel.perf_event_paranoid=1
sudo sysctl kernel.kptr_restrict=0
```

Some locked-down hosts require `kernel.perf_event_paranoid=-1`. Treat that as an explicit security decision.

## Local Linux Command

```bash
perfagent profile run \
  --runtime system \
  --mode ebpf \
  --pid 12345 \
  --duration-seconds 60 \
  --output-dir ./outputs/profiles \
  --output-json ./outputs/profiles/profile-result.json
```

## Docker Compose

The repository includes a privileged Linux profiling service:

```bash
docker compose --profile profiling run --rm perfagent-ebpf \
  profile run \
  --runtime system \
  --mode ebpf \
  --pid 12345 \
  --duration-seconds 60 \
  --output-dir ./outputs/profiles \
  --output-json ./outputs/profiles/profile-result.json
```

The `perfagent-ebpf` service uses:

- `pid: host`
- `privileged: true`
- `SYS_ADMIN`, `SYS_PTRACE`, `PERFMON`, and `BPF` capabilities
- host mounts for `/sys/kernel/debug`, `/sys/kernel/tracing`, `/lib/modules`, and `/usr/src`

Docker Desktop on macOS/Windows profiles the Linux VM, not arbitrary host processes. Use this mode on Linux workers or Kubernetes nodes for real service profiling.

## Kubernetes

Example:

```bash
kubectl apply -f examples/kubernetes/perfagent-ebpf-profiler-job.yaml
kubectl logs job/perfagent-ebpf-profiler
```

The example Job runs with `hostPID: true` and privileged access. In production clusters, bind this to a controlled namespace, service account, admission policy, and node selector. The target PID must be visible from the node PID namespace.

## Evaluation Integration

Capture during load execution:

```bash
perfagent evaluate \
  --service-name payments-api \
  --openapi ./openapi.yaml \
  --target-url http://localhost:8080 \
  --runtime go \
  --slo-p95-ms 500 \
  --slo-error-rate 1 \
  --profile-auto \
  --profile-mode ebpf \
  --profile-pid 12345 \
  --output ./outputs/payments-api
```

The report includes profile artifact links and parsed top functions. PerfAgent still keeps release decisions deterministic: SLOs, capacity, breakpoints, and regressions are calculated from structured metrics, while profile summaries provide supporting evidence.
